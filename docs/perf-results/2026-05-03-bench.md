# Pure-CPU bench subset — 2026-05-03

Captured during the post-fmt verification cycle on commit `c05e8da`
(immediately after the workspace `cargo fmt --all` + composer-cache
guard fix). Four pure-CPU benches in the workspace bench inventory
were run; the rest are infra-bound (Kafka / QUIC) and were not run
in this pass.

## Run context

| Field | Value |
|---|---|
| Host | Windows 11 Home, dev laptop |
| Profile | `bench` (release + debuginfo) |
| Sample count | 20 (criterion `--sample-size 20`) |
| Warm-up | 1 s |
| Measurement | 5 s target |
| Concurrency | A second `cargo test --workspace` was running against the same `target/` for ~2 minutes during the start of this bench run. Absolute timings (and the `change:` deltas) are inflated by ~20-60 % vs single-process baselines on the same machine. **These numbers are NOT a regression signal — they are a snapshot under known load.** |
| Acceptance gates | `multi_partition_blackhole_bench` enforces gate-1 scaling thresholds intended for dedicated Linux hardware. Some FAIL on a Windows dev box; this is expected (per the throughput-validation methodology — hit-the-lid only on real infra). |

## Reproduce

```bash
cd tmp/bench-results
for b in crypto_bench eo2_durability_bench multi_partition_blackhole_bench pipeline_components_bench; do
  cargo bench --bench "$b" --manifest-path ../../Cargo.toml \
    -- --warm-up-time 1 --measurement-time 5 --sample-size 20
done
```

Raw output: `tmp/bench-results/run.log` (24 KB, gitignored).

## multi_partition_blackhole_bench (Run 5 — gate-1 scaling)

Configuration: 100 K events / partition, 256 B payload, batch_size 1024.

### Blackhole sink (Aeon-only path)

| Partitions | Throughput | Ratio vs 1p |
|---|---|---|
| 1 | 1,371,505 events/s | 1.00x |
| 2 | 2,331,497 events/s | 1.70x |
| 4 | 2,866,979 events/s | 2.09x |
| 8 | 3,575,240 events/s | 2.61x |

### FileSink (Aeon + filesystem write)

| Partitions | Throughput | Ratio vs 1p |
|---|---|---|
| 1 | 373,295 events/s | 1.00x |
| 2 | 674,906 events/s | 1.81x |
| 4 | 1,281,468 events/s | 3.43x |
| 8 | 1,694,177 events/s | 4.54x |

### Blackhole vs FileSink gap

| Partitions | FileSink as % of Blackhole |
|---|---|
| 1p | 27.2 % |
| 2p | 28.9 % |
| 4p | 44.7 % |
| 8p | 47.4 % |

### Acceptance gates

| Gate | Threshold | Observed | Verdict |
|---|---|---|---|
| Blackhole 2p | ≥ 1.8x | 1.70x | ❌ FAIL |
| Blackhole 4p | ≥ 3.5x | 2.09x | ❌ FAIL |
| Blackhole 8p | ≥ 6.0x | 2.61x | ❌ FAIL |
| FileSink 2p | ≥ 1.7x | 1.81x | ✅ PASS |
| FileSink 4p | ≥ 3.0x | 3.43x | ✅ PASS |
| FileSink 8p | ≥ 5.0x | 4.54x | ❌ FAIL |
| FileSink/Blackhole gap @ 4p | > 50 % | 44.7 % | ❌ FAIL |

**Caveat:** these gates are calibrated for dedicated Linux hardware
running on local NVMe with no concurrent CPU load. The Windows dev
laptop with a parallel `cargo test` cycle hammering the same target/
will not satisfy them; that is the documented expectation of the
gate-1 methodology (see `feedback_throughput_validation.md` —
"Hit-the-lid: provision infra so it's not the bottleneck"). The
gates apply to Session-B EKS / GCP runs, not local-dev validation.
